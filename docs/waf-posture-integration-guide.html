What Is WAF Posture?

WAF Posture is an optional AstraNull add-on that answers one enterprise question:

Core question

Across every approved web-facing asset, which apps are protected by a WAF/CDN, which are underprotected, which are unprotected — and can we prove the protection actually works?

It extends DDoS readiness into external web control validation. The core platform validates declared target groups with outside probes and inside agents. WAF Posture adds a parallel domain for WAF/CDN coverage, effectiveness, drift detection, and remediation — without requiring cloud or WAF credentials by default.

Architecture

Target Groups → WAF Assets → Safe Probe Jobs + Agent Observations → Posture Snapshots → Findings → Drift Events → Tickets / Reports
Not a separate product

WAF Posture plugs into the existing validation loop — target groups, probes, agents, test runs, and findings. It is feature-gated and disabled by default. When off, /v1/waf/* returns 404 waf_feature_disabled.

Protection Status Model

Protected

WAF/CDN present and intended safe validation is blocked/challenged before origin. Requires product fingerprint + pass on required safe checks + no confirmed bypass.

Underprotected

WAF/CDN present but at least one gap exists: monitor-only signal, stale rules, failed marker, bypass, drift, or missing policy.

Unprotected

No WAF/CDN protection detected where policy requires it. No fingerprint and direct app exposure confirmed.

Unknown

Not enough evidence. Probe blocked, missing agent placement, unsupported vendor, or no customer approval.

Excluded

Customer intentionally excludes from WAF policy. Approved exception with owner, expiry, and scope hash.

Integration: DDoS Readiness Perspective

WAF Posture extends the existing validation loop. It does not replace DDoS readiness — it adds an L7 control-validation dimension on the same infrastructure.

Shared Foundation

Target Group → Targets / URLs + Agents / Canaries → Test Runs → Probe Events + Agent Observations → Verdict / Correlation → Findings

WAF Asset → WAF Validation Run → (same Test Runs) → Posture Snapshot → WAF Findings

Correlation Truth Table

The same inside/outside proof model used for DDoS readiness applies to WAF marker validation:

Probe ResultAgent ResultVerdict
Block / challengeNot observedPass — WAF blocked before origin
Normal app responseObservedFail — monitor-only or missing rule
TimeoutObservedFail — traffic penetrated but did not return
Block / challengeObservedAmbiguous — investigate placement
Normal app responseNot observedInconclusive — wrong target/agent placement or cache

Step-by-Step Integration

  1. Enable the feature — Set ASTRANULL_WAF_POSTURE_ENABLED=1 in your environment.
  2. Use existing target groups — WAF assets are created on top of target groups; they don't replace them.
  3. Position agents — Same agent/canary placement as DDoS readiness. Origin-bypass and marker validation require inside/outside proof.
  4. Create WAF assets — POST /v1/waf/assets with target_group_id, canonical_url, and metadata (criticality, vendor hint, compliance tags).
  5. Customer creates marker rule — Harmless WAF rule matching AstraNull's generated nonce (header, path, query, or user-agent marker).
  6. Run safe checks — Via POST /v1/waf/validations (ad hoc) or POST /v1/waf/validation-plans (batch). Both delegate to testRuns.startTestRun.
  7. Finalize posture — POST /v1/waf/validations/:id/finalize correlates probe + agent events, classifies posture, creates snapshots and findings.
  8. Review in shared surfaces — WAF gaps appear as findings with reason codes alongside existing test-run evidence.

Example: Create a WAF Asset

POST /v1/waf/assets
{
  "target_group_id": "<existing-group>",
  "canonical_url": "https://app.example.com",
  "expected_waf_required": true,
  "expected_vendor_hint": "cloudflare",
  "business_criticality": "high",
  "traffic_tier": "high"
}

Example: Run Combined Validation

POST /v1/waf/validations
{
  "waf_asset_id": "<asset-id>",
  "modes": ["fingerprint", "marker", "origin_bypass", "combined"]
}

Integration: Drift Posture Perspective

Drift is a longitudinal layer on top of point-in-time posture. It answers: "Did protection weaken since we last proved it?"

Drift Architecture

Approved Baseline → Scheduled Validation / Connector Poll → New Snapshot → Compare → Drift Event → Retest → Evidence Closure

Drift Types

Drift TypeDetectionSeverity
fingerprint_lostPreviously detected WAF no longer detectedHigh
marker_failedMarker previously blocked, now passes throughHigh
origin_bypass_newDirect origin path now reachableCritical
mode_changeBlocking became monitor/detect/log-onlyHigh
vendor_changedProduct fingerprint changed without approvalMedium
rule_count_changeActive rule count dropped (connector-only)Medium
threshold_changeRate/scoring threshold relaxed (connector-only)Medium
scenario_regressionScenario family pass rate dropsHigh

Drift Event Lifecycle

open → acknowledged → remediation_started → retest_pending → resolved

remediation_started → accepted_risk or false_positive

Step-by-Step Drift Integration

  1. Establish a baseline — POST /v1/waf/assets/:id/baselines then POST /v1/waf/baselines/:id/approve after first successful validation.
  2. Schedule recurring checks — Validation plans via cron (npm run waf:orchestrator:runner), drift scans (npm run waf:drift:runner), or connector polls (npm run connector:poll:runner).
  3. Monitor drift events — GET /v1/waf/drift-events or the drift queue on #waf-posture in the UI.
  4. Triage in UI — Patch status (acknowledged → remediation_started → retest_pending). Request retest via POST /v1/waf/drift-events/:id/retest.
  5. Execute retest — POST /v1/waf/retests/:id/execute delegates safe test runs (one per tick in signed-worker mode).
  6. Close with evidence — POST /v1/waf/retests/:id/complete closes only when verdict evidence is finalized. UI clicks alone do not close drift.
  7. Wire into ops toolchain — SIEM events, Jira/ServiceNow tickets, Slack alerts, and audit exports.
Closure requires evidence

Drift resolves only on: retest passes with finalized verdict evidence, approved new baseline, or approved risk exception with expiry. Plan completed status means jobs were delegated — not that posture is fixed.

Summary

From DDoS Readiness

WAF Posture is an L7 control-validation extension of your existing target-group → probe → agent → finding loop. Add WAF assets, run safe catalog checks through the same test-run infrastructure, and get posture snapshots alongside readiness evidence.

From Drift Posture

Add baselines, scheduled re-validation, snapshot comparison, and a retest closure loop. Drift events are the operational signal; retest completion with verdict evidence is the only trustworthy close.

Key caveats

Many connector, ticketing, and production-orchestration paths are still open. Outside-in validation has known blind spots (agent placement, allowlisting, CDN vs WAF). Validation plan completion does not mean posture is fixed — always follow through to snapshot, finding, and drift closure.
