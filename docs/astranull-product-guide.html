AstraNull Features & Functionality

A no-access-first DDoS readiness validation platform. Customers declare target groups, install outbound-only agents, and AstraNull correlates safe external probes with internal observations to produce evidence-backed verdicts, findings, and reports.

Product Promise & Principles

What AstraNull proves

DDoS readiness for customer-declared targets without requiring cloud credentials or automatic IP inventory discovery.

Core Principles

PrincipleMeaning
No-access-firstCore value works without AWS/GCP/Azure/CDN/WAF credentials.
Customer-declared scopeCustomers explicitly define FQDNs, URLs, IPs, ports — no mandatory discovery.
Agent-observed truthStrong verdicts require inside/outside correlation via outbound-only agents.
Outbound-only controlAgents call AstraNull over HTTPS; no inbound management ports required.
SOC gates riskHigh-scale simulations are request-only; SOC approves, executes, and stops them.
Evidence-first UXEvery page answers: what was tested, what happened, what proof exists, what to do next.
Safe by defaultDefault checks are low-volume, bounded, and non-disruptive.

What AstraNull Does Not Do

ExcludedWhy
Automatic IP/cloud inventory discoveryConflicts with no-access-first; customer declares scope.
Self-service DDoS attack generationUnsafe and legally risky; high-scale is SOC-gated.
Replacement for DDoS providersValidates readiness; does not replace Cloudflare, Akamai, Shield, etc.
Mandatory cloud config auditRequires credentials; optional connector mode only.
Autonomous WAF rule deploymentHuman approval and change control required (WAF add-on).

End-to-End Flows

Flow A — Customer Onboarding

Sign up→ Create environment→ Target group→ Declare targets→ Bootstrap token→ Install agent→ Verify heartbeat→ First safe run

Install choices: Linux one-line, Debian/Ubuntu (.deb), RHEL/Amazon Linux (.rpm), generic tarball, Docker, Kubernetes (Helm), packet mirror collector.

Flow B — Safe Validation Run

User clicks Run→ Planner validates scope→ Agent arms observation→ Probe sends nonce→ Agent uploads observation→ Correlation→ Verdict + Finding
StateWhat Happens
PlannedSelect enabled checks, targets, safety caps, probe regions.
Agent ArmedObservation window opened before probe starts.
ProbingSigned probe worker sends bounded low-rate labeled probes.
CollectingBackend waits for probe and agent events.
CorrelatingMatches by test_id, nonce, target, time window.
VerdictedProtected, Bypassable, Inconclusive, Misplaced Agent, Partial.
ReportedFindings, evidence timeline, notifications.

Flow C — High-Scale Request (SOC-Gated)

Customer requests→ Authorization pack→ SOC review→ Schedule + execute→ Kill switch / stop→ Post-test report→ Closed

Customers can request high-scale tests but cannot start, execute, or stop them. SOC owns approval, scheduling, execution, and closure.

Validation & Detection

How Proof Works

Outside Probe → CDN / WAF / Scrubbing → Origin → Inside Agent

Correlation matches probe nonce with agent observation (or non-observation) to determine whether traffic was blocked, penetrated, or bypassed.

Vector Families

FamilyExample ChecksWho Runs
Origin bypassDirect IP, Host/SNI bypass, stale DNS, CDN bypassCustomer (safe)
L3/L4TCP connect, forbidden port, IPv6, protocol exposureCustomer (safe)
DNSResolver exposure, NXDOMAIN, DNSSEC, failover, AXFRCustomer (safe)
L7 / APIHTTP methods, WAF markers, rate limits, expensive endpoints, GraphQLCustomer (safe/controlled)
TLS / ConnectionHandshake, slow headers, connection timeoutsCustomer (safe)
ProtocolsHTTP/2 rapid reset, HTTP/3/QUIC, gRPC, WebSocketCustomer (safe) / SOC
WAF postureFingerprint, marker rule, origin bypass, low-rate limitCustomer (WAF add-on)
High-scale DDoSMulti-vector drills, connection exhaustion, provider telemetrySOC only

Check Risk Classification

ClassDescriptionExecution Owner
SafeSingle/few labeled probes; non-disruptive.Customer engineer
ControlledBounded repeated probes with explicit caps.Customer admin/engineer
SOC-gatedHigh-scale or disruptive potential.AstraNull SOC only
ProhibitedExploit payloads, unmanaged attack generation.Never customer-runnable

Verdict Types

VerdictMeaning
ProtectedTraffic blocked/challenged before origin; agent did not observe nonce.
Bypassable / PenetratedTraffic reached origin; agent observed nonce or direct path confirmed.
InconclusiveInsufficient evidence — wrong placement, missing agent, probe blocked.
Misplaced AgentAgent cannot observe traffic for this target group.
PartialSome targets passed, others failed within same run.

Guardrails on Every Safe Run

  • Target group safety policy and safe windows
  • Tenant rate budget and per-group cooldown
  • Per-run event caps and concurrent-run block
  • Signed probe jobs with max_requests / timeout_ms caps
  • Tenant kill switch auto-cancels active safe runs
  • Agent observations reject raw packet/body/header fields

Findings Workflow

Findings are created from bypassable verdicts, missing controls, misplaced agents, or stale critical evidence. Workflow: open → assigned → remediation → retest → closed (or accept-risk with owner approval).
